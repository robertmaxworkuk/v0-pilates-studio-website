# GITHUB ISSUES GUIDE

> Версия: 1.2  
> Назначение: единый стандарт работы с GitHub Issues для всех ИИ-агентов разработки в проекте.

## 1) Цель документа

Этот документ задаёт обязательный, прозрачный и проверяемый процесс работы с задачами (Issues) в GitHub, чтобы:
- видеть фактический прогресс проекта;
- избегать потери контекста между чатами и агентами;
- связывать код, обсуждения и этапы проекта;
- поддерживать дисциплину планирования и исполнения.

---

## 2) Когда использовать и как технически работать

Передавайте этот файл **в начале каждого нового чата** с агентом разработки.

Агент обязан:
1. Ознакомиться с текущими открытыми issues и milestone (используя интегрированные MCP-инструменты: `search_issues`, `list_issues`, `list_milestones`).
2. Уточнить, с какой задачей он работает в данном чате.
3. Читать, создавать и обновлять задачи через соответствующие API (например, `issue_read`, `issue_write`, `add_issue_comment`).
4. После завершения работы зафиксировать результат в соответствующем issue.

---

## 3) Базовые принципы (обязательно)

1. **Одна проблема = один issue**. Не смешивать разные задачи в одном issue.
2. **Issue создаётся до начала заметной работы** (код, архитектурное решение, изменение UX, инфраструктура).
3. **Каждый PR должен ссылаться на issue** (`Closes #123`, `Refs #123`).
4. **Ветки (branches) должны содержать номер issue** (например: `feature/123-add-faq`, `bugfix/124-header`).
5. **Каждое изменение статуса должно быть отражено в issue-комментарии**.
6. **Нельзя закрывать issue без проверки критериев готовности (DoD)**.
7. **Источник истины по прогрессу — GitHub Issues + Milestones**, а не чат.

---

## 4) Стандарт жизненного цикла issue

Используйте следующий workflow:

1. **Backlog** — идея зафиксирована, но не взята в работу.
2. **Ready** — задача уточнена, есть критерии приёмки.
3. **In Progress** — идёт активная работа.
4. **In Review** — есть PR, ждём ревью/проверки.
5. **Blocked** — есть блокер (зависимость, баг, доступы и т.п.).
6. **Done** — принято, закрыто, результат зафиксирован.

Если у вас включён Projects board, этим состояниям должны соответствовать колонки/статусы.

---

## 5) Обязательный шаблон issue

При создании нового issue используйте структуру:

### Title
Коротко, конкретно, в формате действия.

**Пример:** `Добавить секцию FAQ на лендинг`

### Description
- **Контекст:** зачем это нужно.
- **Проблема:** что сейчас не так.
- **Ожидаемый результат:** что должно быть после завершения.

### Acceptance Criteria (чекбоксами)
- [ ] Критерий 1 (измеримый)
- [ ] Критерий 2 (измеримый)
- [ ] Критерий 3 (измеримый)

### Декомпозиция (для крупных задач)
Если задача объемная (занимает больше 1-2 сессий), ИИ-агент обязан декомпозировать её на подзадачи в виде дополнительных чек-листов или через создание дочерних Sub-issues.

### Scope
- **In scope:** что делаем.
- **Out of scope:** что точно не делаем.

### Links / Dependencies
- Связанные issues/PR/документы.
- Блокирующие зависимости.

### Meta
- **Labels:** `type:*`, `priority:*`, `area:*` (см. раздел ниже)
- **Assignee:** ответственный
- **Milestone:** этап проекта

---

## 6) Единые правила для labels

Все системные метки уже заведены нами в интерфейс GitHub! Используй их строго по стандарту:

### По типу
- `type:feature` — новая функциональность
- `type:bug` — ошибка
- `type:chore` — рутина, обновления зависимостей
- `type:docs` — работа с документацией
- `type:refactor` — рефакторинг без изменения логики
- `type:research` — исследование (R&D)

### По приоритету
- `priority:P0` — критично, блокирует релиз/бизнес
- `priority:P1` — высокий
- `priority:P2` — средний
- `priority:P3` — низкий

### По области
- `area:frontend` — клиентская часть
- `area:backend` — серверная часть
- `area:design` — дизайн/UX
- `area:infra` — инфраструктура, CI/CD, настройки
- `area:content` — тексты, переводы, медиа
- `area:analytics` — метрики и аналитика

### По состоянию
- `status:blocked` (ставится только при явном блокере)
- Остальные (дефолтные github): `duplicate`, `invalid`, `wontfix`, `question`

---

## 7) Требования к обновлениям issue во время работы

Агент должен оставлять короткие, объективные апдейты (используя API комментирования `add_issue_comment`):

### При старте
- «Взял в работу #123, ветка: `feature/123-название`, план: ...»
- **Обязательно проставь/проверь метки (labels)** задачи по типу, приоритету и области (например: `type:feature`, `priority:P1`, `area:frontend`).

### При промежуточном прогрессе
- Что сделано фактически.
- Что осталось.
- Есть ли риски/изменение оценки.

### При блокировке
- Что блокирует.
- Какой следующий шаг для разблокировки.
- Кто нужен для решения.

### Перед закрытием
- Ссылка на PR.
- Результат проверки acceptance criteria.
- Что вошло/не вошло в объём.

---

## 8) Правила привязки PR к issue

В каждом PR обязательно:

1. Ссылка на issue в описании.
2. Явный статус связи:
   - `Closes #123` — если полностью закрывает issue;
   - `Refs #123` — если частично или связанно.
3. Краткий список: «что сделано / что не сделано».
4. Если PR частичный — в issue оставить комментарий с остатком работ.
5. **После успешного слияния (merge) PR текущая ветка должна быть удалена как удалённо на сервере, так и локально.**

---

## 9) Definition of Done (DoD) для закрытия issue

Issue можно закрыть только если:
- [ ] Все acceptance criteria отмечены выполненными.
- [ ] Код в `main` (или целевой ветке) через связанный PR.
- [ ] Ветка git удалена (обязательно после слияния PR!).
- [ ] Прошли обязательные проверки (линтеры/тесты/ревью — по правилам проекта).
- [ ] Нет открытых блокеров по этой задаче.
- [ ] В issue есть финальный комментарий с итогом и ссылками.

Если хотя бы один пункт не выполнен — issue не закрывать.

---

## 10) Milestones и этапы проекта

- Каждый issue должен быть привязан к milestone (если это не мелкий ad-hoc).
- Milestone соответствует этапу (например: `M1 — MVP`, `M2 — Payments`, `M3 — Growth`).
- Прогресс этапа считается по закрытым issues в milestone.

Рекомендуется еженедельно:
- пересматривать backlog;
- переприоритизировать `P0/P1`;
- переносить незавершённые задачи в актуальный milestone с комментарием причины.

---

## 11) Команда агенту (шаблон для вставки в новый чат)

Скопируйте и вставляйте агенту в начале работы. Это задает строгий технический контекст.

```md
Работай строго по GITHUB_ISSUES_GUIDE.md.
Текущий репозиторий: robertmaxworkuk/v0-pilates-studio-website

Обязательные шаги:
1) Используя MCP (search_issues/list_issues), определи, с каким issue ты работаешь (или создай новый по шаблону).
2) Обновляй issue (add_issue_comment) при старте, прогрессе, блокере и завершении. Всегда проверяй и обновляй метки (labels) по правилам гайда.
3) Создавай ветки git с номером issue (feature/123-short-name).
4) Каждый PR должен ссылаться на issue (Closes/Refs).
5) Перед закрытием issue проверь Definition of Done.
6) В финальном отчёте чата укажи: номер issue, статус, что сделано, что осталось.
```

---

## 12) Анти-паттерны (запрещено)

- Делать работу без issue, «потому что мелочь».
- Создавать ветки без привязки к номеру issue.
- Игнорировать встроенные инструменты интеграции с GitHub (MCP).
- Закрывать issue по факту открытия PR, а не по факту выполнения критериев.
- Смешивать багфикс, новую фичу и рефакторинг в одном issue без явного основания.
- Оставлять issue без статуса и без ответственного.
- Не фиксировать блокеры.
- Оставлять старые ветки после слияния PR (надо всегда удалять).

---

## 13) Минимальный стандарт качества issue

Issue считается качественным, если:
- понятен контекст и ожидаемый результат;
- есть измеримые acceptance criteria;
- декомпозировано, если объем работы велик;
- указан приоритет и область;
- есть ответственный и milestone;
- по комментариям можно восстановить историю выполнения.

---

## 14) Роли и ответственность

- **Инициатор задачи:** корректно формулирует проблему и ценность.
- **Исполнитель (человек/ИИ-агент):** поддерживает актуальный статус, ведет ветки правильно и оставляет прозрачные апдейты.
- **Ревьюер/лид:** контролирует качество критериев, приоритет и корректность закрытия.
